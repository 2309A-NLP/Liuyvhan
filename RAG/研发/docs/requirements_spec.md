# 需求规格说明书

## 1. 项目背景

本项目旨在实现一个基于 RAG 的角色扮演式智能对话系统。系统通过“角色设定 + 知识库检索 + 记忆管理 + 大模型生成”的方式，为用户提供多角色、多轮次、具备上下文感知能力的问答体验。

项目当前以本地开发和学习验证为主，但整体结构已经预留出真实 Redis、Milvus、MySQL、Embedding 模型和 Rerank 模型的扩展边界。

## 2. 建设目标

- 实现一个可运行的角色扮演式对话系统
- 支持多个角色分别进行对话
- 支持每个角色基于独立知识库进行检索增强回答
- 支持短期记忆和长期记忆分层存储
- 支持流式输出，提升交互体验
- 支持本地一键启动 Redis、Milvus 和应用服务
- 为后续替换为更真实的生产级组件保留接口边界

## 3. 用户角色

### 3.1 普通用户

能力包括：

- 登录或游客进入系统
- 选择角色
- 发起新会话
- 与角色进行多轮对话
- 查看当前会话历史

### 3.2 管理或开发者

能力包括：

- 初始化数据库和知识库
- 重新加载角色与知识种子数据
- 检查服务健康状态
- 查看日志，判断 Redis 与 Milvus 是否真正接入

## 4. 业务范围

系统当前覆盖以下业务范围：

- 首页展示与工作台入口
- 角色列表展示
- 用户创建与身份进入
- 会话创建与切换
- 角色扮演式问答
- 短期记忆管理
- 长期记忆存储与召回
- 角色知识库检索
- 知识库重载
- 服务健康检查

## 5. 功能需求

### 5.1 首页功能

- 展示系统状态
- 展示角色数量与存储模式
- 提供进入工作台入口

### 5.2 工作台功能

- 展示账号信息
- 展示角色入口
- 点击角色自动新建对话
- 展示当前会话列表
- 展示消息流
- 展示知识引用结果

### 5.3 用户管理功能

- 支持游客快速进入
- 支持具名登录
- 支持创建或覆盖同一用户 ID 的信息

### 5.4 角色管理功能

- 系统启动时加载角色种子数据
- 支持查询全部角色列表
- 每个角色拥有独立设定、语气、规则和提示词模板

### 5.5 对话功能

- 支持普通聊天
- 支持流式聊天
- 支持多轮上下文对话
- 支持按角色维度进行知识检索
- 支持把检索结果作为引用返回前端

### 5.6 短期记忆功能

- 按会话保存最近若干轮消息
- 用户消息与助手消息都要写入短期记忆
- 在下一轮对话中读取短期记忆作为上下文

### 5.7 长期记忆功能

- 从用户消息中抽取偏好、身份、目标等长期信息
- 将长期信息向量化后写入长期记忆集合
- 在后续会话中按用户维度召回长期记忆

### 5.8 知识库功能

- 支持角色知识文档导入
- 支持文本切块
- 支持向量化
- 支持写入 Milvus
- 支持按角色过滤召回
- 支持手动触发知识库重载

### 5.9 健康检查功能

- 返回服务状态
- 返回当前大模型提供方
- 返回当前向量化后端
- 返回当前存储模式

### 5.10 日志功能

- 记录 Redis 连接状态
- 记录 Milvus 连接状态
- 记录集合初始化与加载
- 记录知识块写入
- 记录长期记忆写入
- 记录检索行为

## 6. 业务流程

### 6.1 对话主流程

1. 用户进入工作台。
2. 用户选择一个角色。
3. 系统自动创建一个新会话。
4. 用户输入消息并发起请求。
5. 系统读取角色信息。
6. 系统读取当前会话的短期记忆。
7. 系统检索用户长期记忆。
8. 系统按角色过滤知识库并执行检索。
9. 系统对候选知识片段重排序。
10. 系统组装角色提示词、短期记忆、长期记忆和引用知识。
11. 系统调用大模型生成回答。
12. 系统将回答返回前端。
13. 系统把本轮对话写入短期记忆。
14. 如果命中长期记忆规则，则写入长期记忆。

### 6.2 知识库重载流程

1. 管理者触发知识库重载接口。
2. 系统读取角色种子文件。
3. 系统读取知识种子文件。
4. 系统对知识文档切块。
5. 系统对知识块向量化。
6. 系统写入角色知识库集合。
7. 系统返回写入结果。

## 7. 业务规则

- 一个会话在同一时刻只绑定一个角色
- 知识检索必须带 `role_id` 过滤，避免角色知识串库
- 短期记忆默认保留最近若干轮对话
- 长期记忆按 `user_id` 维度检索
- 长期记忆只对符合规则的用户消息进行写入
- 心理、法律、财富类角色应具有边界提示，不直接替代线下专业服务
- 如果 Redis 或 Milvus 不可用，系统允许降级到本地文件模式

## 8. 非功能需求

### 8.1 可用性

- 系统应能在本地环境稳定启动
- 支持 Windows + WSL + Docker 的组合运行方式
- 支持通过启动脚本完成一键启动

### 8.2 可维护性

- 模块职责清晰
- 数据访问层、业务层、模型层解耦
- 后续替换具体实现时，不应大规模改动业务逻辑

### 8.3 可扩展性

- 支持新增角色
- 支持替换 Embedding 模型
- 支持替换 Rerank 模型
- 支持替换 SQLite 为 MySQL
- 支持替换本地开发结构为服务化部署

### 8.4 可观测性

- 关键链路具备日志输出
- 可通过 `/health` 快速判断当前运行模式
- 可通过日志确认 Redis 与 Milvus 是否实际生效

### 8.5 性能

- 支持基础流式回答体验
- 检索和记忆查询应在单轮对话中及时完成
- 本地环境下优先保证可运行，再逐步优化效果与速度

## 10. 技术约束

- 当前应用主入口为 FastAPI
- 当前基础存储为 Redis、Milvus、SQLite
- 当前 Embedding 采用轻量 HashingVectorizer
- 当前 Rerank 采用启发式方案
- 当前 LLM 通过 OpenAI Compatible API 调用
- 当前启动方式依赖 WSL 内 Docker 环境

## 11. 验收标准

- 服务能够正常启动
- 首页和工作台能够正常访问
- `/roles` 能返回角色列表
- `/users` 能创建用户
- `/chat` 与 `/chat/stream` 能返回角色回答
- `/sessions/{session_id}/history` 能返回短期记忆
- `/knowledge/reload` 能重新加载知识库
- `/health` 能返回当前运行模式
- 日志中能够看到 Redis 与 Milvus 连接成功信息

## 12. 当前实现现状

当前项目已经具备以下实际能力：

- Redis 已接入并可用于短期记忆
- Milvus 已接入并可用于知识库和长期记忆
- 工作台支持角色点击后自动创建新会话
- 聊天支持流式返回
- 项目支持一键启动脚本

## 13. 阶段化实现说明

当前版本采用的是“先跑通完整链路，再逐步替换为标准实现”的策略，而不是一开始就把所有模块都升级到最终形态。这样设计的原因包括：

- 第一阶段更重视系统是否能够闭环运行
- 本地环境中已经包含 Windows、WSL、Docker、Redis、Milvus 和外部 LLM API，多组件同时标准化会显著提高排错成本
- 先完成轻量实现，有助于验证架构边界是否设计正确
- 先保证“可运行、可演示、可排错”，再做模型级升级，更符合本地开发节奏

## 14. 后续演进方向

推荐升级顺序如下：

1. 优先完成工程与存储标准化，例如 MySQL、日志、测试、前端一致性和检索策略优化。
2. 再优化记忆规则、Query Rewrite 和召回质量。
3. 之后谨慎尝试真实 BGE-M3。
4. 最后再考虑真实 BGE-Reranker。

原因是工程层升级通常对本机压力较小，而真实模型接入会显著增加资源消耗。
